## 2. 핵심 설계 결정

### 2.1 analyze-tab.html 의 역할
panel.html 에는 HTML include 메커니즘이 없고, 안전 규칙(`chrome-extension-security §3-1`)으로 raw HTML 의 속성 기반 삽입 경로가 금지된다. 따라서 `.html` 파일은 **참조본**으로만 두고, 실제 DOM 은 `analyze-tab.js` 의 `buildAnalyzeTabDOM()` 이 `createEl()` 호출로 등가 구조를 생성한다. 두 파일은 1:1 마크업 동치를 유지한다(파일 상단 주석 명시).

### 2.2 마운트 훅 패턴
`panel.js` 에 `tabActivators` 맵을 도입해 탭 ID → async 마운트 함수 매핑.
- analyze 탭 활성화 시 `import("./tabs/analyze-tab.js")` 로 lazy-load 후 `mountAnalyzeTab(panelEl)` 호출.
- `mountAnalyzeTab` 은 idempotent(`mounted` 플래그) — 탭을 여러 번 들락거려도 1회만 마운트.
- 다른 탭(benchmark/generate/mypage)은 후속 Phase 에서 같은 패턴으로 등록.
- 마운트 실패해도 탭 전환 자체는 막지 않음(catch + console.error).

### 2.3 메시지 흐름
```
[사용자: 분석 시작 클릭]
  ↓
chrome.tabs.query({ active: true, currentWindow: true })
  ↓
URL 검증: NAVER_BLOG_HOST_RE = /^(?:m\.)?(?:[a-z0-9-]+\.)?blog\.naver\.com$/i + https only
  ↓
chrome.runtime.sendMessage({ action: 'analyze.post', payload: { url } })
  ↓ (service-worker.js → handlers/analyze-handler.js — 후속 Phase)
응답: { ok: true, data: {...} } | { ok: false, error }
  ↓
정규화(normalizeAnalyzeResult) → renderSummary + renderSections(score-card 카드 N개)
```

## 3. 보안·안전 체크포인트

| 항목 | 적용 방식 |
|---|---|
| 위험 DOM 속성 직접 할당 0건 | 모든 DOM 조작은 `dom-safe.js` 의 `createEl`/`safeText`/`clearAndAppend` 만 사용 |
| 외부 데이터 textContent 경로 | 블로그 제목·URL·권장사항 문자열은 `createEl(tag, attrs, [string])` 의 children 슬롯 → 자동 `createTextNode` |
| 동적 코드 실행 | 없음 (정적 import 만 사용, lazy `import()` 도 정적 경로) |
| sender 출처 검증 | service-worker 측 책임(후속 Phase). UI 는 송신자라서 `sendMessage` callback 의 `chrome.runtime.lastError` 체크만 수행 |
| URL 검증 | `new URL()` + 정규식 + `https:` 강제. 네이버 블로그 외 거부 메시지 표시 |
| API 키 노출 | 분석 탭은 SDK 호출 없음. 키 관련 코드 0건 |
| Tailwind CDN | 미사용 (panel.css 만 확장) |
| innerHTML 류 | 0건 — `check-dom-unsafe.sh` PASS 로 확인 |
