## 아키텍처

### 이중 격리 (host page 오염 차단)

```
blog.naver.com DOM
└─ <div id="bbm-root"> ← host page 에 추가된 유일한 엘리먼트
   └─ #shadow-root (closed)  ← CSS/JS 격리막 1
      ├─ <link rel="stylesheet" href="content/content.css">
      └─ <aside id="bbm-sidebar">
         ├─ <button id="bbm-sidebar-toggle">
         └─ <iframe id="bbm-sidebar-frame"
                    src="chrome-extension://<id>/content/sidebar.html"> ← 격리막 2
                                                                           (extension origin)
```

- **격리막 1 (Shadow DOM closed)**: host page 의 naver CSS 가 사이드바에 적용되지 않음. host JS가 `document.querySelector('#bbm-sidebar')` 로 내부 DOM 조회 불가.
- **격리막 2 (iframe extension origin)**: 사이드바 JS 가 `chrome.*` API 를 extension 권한으로 호출 가능. host JS 와 JS context 완전 분리.

### 메시지 흐름

```
[에디터 input]
     │  800ms debounce
     ▼
[analyzer-bridge.js]
  └─ extractEditorContent() (host DOM 읽기 전용)
  └─ chrome.runtime.sendMessage({ action:'analyze.post', payload:{title,body} })
     │
     ▼
[background/service-worker.js → analyze-handler (Phase 3.1)]
     │
     ▼ response { ok, data }
     │
[analyzer-bridge.js]
  └─ iframe.contentWindow.postMessage({type:'BBM_ANALYSIS', payload}, EXTENSION_ORIGIN)
     │
     ▼
[sidebar.js (iframe 내부)]
  └─ origin 검증 → renderScore/renderBasic/renderHook/renderSuggestions
```

### URL 게이트

`sidebar-injector.js`가 다음 URL에서만 주입(manifest matches는 네이버 블로그 전역이므로 JS 에서 2차 필터):
- `/PostWriteForm\.naver/i`
- `/GoBlogWrite\.naver/i`
- `/\/post\/write/i`
- `/editor/i`

### Debounce

`analyzer-bridge.js` 상수 `DEBOUNCE_MS = 800`. `input`/`keyup`/`MutationObserver(characterData+childList)` 세 소스 모두 동일 debounce 타이머 재설정. 연속 타이핑 중 분석 요청 0건, 800ms 휴지 후 1건 발송.

## 의존성 / 후속 조치 (manifest 및 타 에이전트)

### [BLOCKER] manifest.json 변경 필요 (기획자/백엔드 담당)

본 작업 범위에서 manifest 수정 금지 규칙 준수. 다음 항목이 **반드시 manifest.json 에 추가**되어야 content script가 동작합니다.

1. `content_scripts` 블록 등록 — 체크리스트상 "이미 등록됨" 지시였으나 현 manifest.json:1-35 에 `content_scripts` 필드 **부재**. 추가안:
   ```json
   "content_scripts": [{
     "matches": [
       "https://blog.naver.com/*",
       "https://*.blog.naver.com/*",
       "https://m.blog.naver.com/*"
     ],
     "js": [
       "content/sidebar-injector.js",
       "content/analyzer-bridge.js"
     ],
     "run_at": "document_idle"
   }]
   ```
2. `web_accessible_resources` — iframe src 와 동적 import 대상:
   ```json
   "web_accessible_resources": [{
     "resources": [
       "content/sidebar.html",
       "content/sidebar.js",
       "content/content.css",
       "lib/dom-safe.js"
     ],
     "matches": [
       "https://blog.naver.com/*",
       "https://*.blog.naver.com/*",
       "https://m.blog.naver.com/*"
     ]
   }]
   ```

### [BLOCKER] background/handlers/analyze-handler.js (Phase 3.1 담당)

`analyze.post` action 을 처리하는 핸들러가 없으면 `analyzer-bridge.js`의 `sendMessage`가 service-worker 의 "Unknown action" 응답으로 실패(sidebar 상태 → `error/unknown_error`). Phase 3.1 에서 다음 shape 계약을 준수해야 합니다:

```ts
// 요청
{ action: 'analyze.post', payload: { title: string, body: string, source: string, at: number } }
// 응답 (성공)
{ ok: true, data: {
  score: number,
  scoreHint?: string,
  basic?: { charCount, wordCount, paragraphCount, imageCount?, linkCount?, readingTimeMin? },
  metrics?: {...},             // basic 별칭
  hook?: string | { text, score },
  suggestions?: string[],
}}
// 응답 (실패)
{ ok: false, error: 'unauthenticated' | 'rate_limited' | string }
```

### [HIGH] SmartEditor iframe 접근

네이버 SmartEditor 는 본문을 iframe(#mainFrame 등) 에 띄웁니다. 현재 `extractEditorContent()` 는 same-origin 접근을 시도하되 실패 시 조용히 폴백합니다. 실 사용 시 DOM 트리 검증 필요 — Phase 3.1 `extractor.js` 와 셀렉터 전략을 합치는 것이 권장됩니다.

### [MEDIUM] i18n

하드코딩 한국어("실시간 분석", "대기 중", "분석 중…" 등) — Phase 10 에서 `chrome.i18n.getMessage()` 로 이관.

### [MEDIUM] iframe 스타일 중복

`content.css` 의 `:host` 디자인 토큰은 Shadow DOM 한정이므로 iframe 문서는 사용 불가. 현재 `sidebar.html` 내 `<style>` 블록에 최소 스타일을 중복 정의. 토큰 통일은 별도 `sidebar.css` 분리 + iframe/Shadow 양쪽에서 `<link>` 로 참조하는 방식으로 후속 리팩터 가능.

## Summary

- **생성 파일**: 5개(지정 4 + 보조 `sidebar.js` 1)
- **host page 건드린 엘리먼트**: `<div id="bbm-root">` 1개 (Shadow DOM 컨테이너)
- **debounce**: 800ms 단일 타이머, input/keyup/MutationObserver 모두 동일 스케줄러 재사용
- **dom-safe 강제**: iframe 측은 정적 import, content script 측은 `chrome.runtime.getURL` + 동적 import
- **manifest 선행 조건**: `content_scripts` + `web_accessible_resources` 두 블록 추가 필요(현 manifest 부재)
- **Phase 3.1 의존**: `analyze.post` handler 계약 준수 필요(본 요약에 shape 명시)
