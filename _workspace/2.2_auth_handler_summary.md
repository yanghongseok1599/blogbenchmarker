## 액션 목록 (라우팅 맵)

| 액션 | payload | 반환 data |
|---|---|---|
| `auth.login` | `{ email, password }` | `SessionSummary` (userId/email/displayName/plan/isAdmin/emailConfirmed/expiresAt) |
| `auth.signup` | `{ email, password, displayName? }` | `{ userId, email, needsEmailConfirmation }` |
| `auth.logout` | — | `{ redirectTo: chrome.runtime.getURL('auth/login.html') }` |
| `auth.resetPassword` | `{ email, redirectTo? }` | `{ sent: true, email }` |
| `auth.getSession` | — | `{ loggedIn: false }` 또는 `{ loggedIn: true, ...SessionSummary }` |
| `auth.onAuthChange` | — | `{ subscribed: true, alreadyBound? }` — SW 구독 멱등 등록 |

모든 응답은 router 에서 `{ ok: true, data }` / `{ ok: false, error }` 로 래핑된다.

## 자동 재로그인 방지 체인 (로그아웃 순서)

```text
authHandler.logout()
  ├─ 1) chrome.storage.local.set({ __intentional_logout: true })
  ├─ 2) supabase.auth.signOut()            // 네트워크 실패해도 catch 후 진행
  ├─ 3) chrome.storage.local.clear()       // 플래그 포함 모든 로컬 상태 제거
  └─ 4) chrome.storage.session.clear()     // 지원 시 보조 정리
       → return { redirectTo: 'auth/login.html' }

authHandler.onAuthChange() 내부 리스너
  ├─ SDK 가 SIGNED_IN / TOKEN_REFRESHED 등을 쏟아낼 때
  ├─ __intentional_logout === true && event !== 'SIGNED_OUT'  →  broadcast 스킵
  └─ 그 외 → chrome.runtime.sendMessage({ type: 'auth:change', event, session })
```

**핵심 안전장치:**
- 로그아웃 시 플래그를 **signOut 호출 전** 에 설정 → SDK 가 어떤 순서로 이벤트를 발화해도 리스너가 해석 가능.
- `storage.clear()` 는 `signOut` 결과와 무관하게 실행 → 서버 오류로 잔존 세션 방지.
- 브로드캐스트에는 `access_token / refresh_token` 을 포함하지 않음(`userId / email / expiresAt` 3개 필드만).

## 주요 설계 결정

1. **핸들러 반환 = raw data**: router 가 래핑하므로 핸들러는 throw 또는 data 반환만. 이중 래핑 방지.
2. **`toSummary` 단일 진입점**: login / getSession 이 동일한 `SessionSummary` shape 을 반환해 호출 측 분기 감소. qa_report_2.1 J-1 의 계약 `{ userId, email, plan }` 을 상위 호환으로 확장 (`displayName/isAdmin/emailConfirmed/expiresAt` 추가).
3. **`loadProfileSafe`**: signup 직후 `handle_new_user()` 트리거 race 대응. 프로필 미생성/RLS 차단 시 null 반환으로 로그인 흐름을 막지 않음. qa_report_2.1 J-3 의 race 우려 해소.
4. **SW 부팅 훅(`bootTasks`)**: 개별 확장 페이지가 `auth.onAuthChange` 를 호출하지 않아도 SW 시작 시 자동 구독. 브로드캐스트는 `chrome.runtime.sendMessage` 로 — 수신자 부재 시 `.catch(() => {})` 로 조용히 무시.
5. **`__resetAuthChangeBinding` export**: 테스트에서 리스너 상태 초기화 가능. 운영 코드에선 호출 안 함.

## 사용 예제

### sidepanel 또는 mypage 에서 로그인

```js
// 사이드패널·마이페이지 등 확장 페이지 내부
async function doLogin(email, password) {
  const res = await chrome.runtime.sendMessage({
    action: 'auth.login',
    payload: { email, password },
  })
  if (!res.ok) {
    showError(res.error)   // 이미 한국어로 매핑된 상태
    return
  }
  const { userId, plan, isAdmin } = res.data
  renderLoggedIn(res.data)
}
```

### 로그아웃 버튼

```js
document.getElementById('logout').addEventListener('click', async () => {
  const res = await chrome.runtime.sendMessage({ action: 'auth.logout' })
  if (res.ok) location.href = res.data.redirectTo
})
```

### 세션 변화 수신 (사이드패널)

```js
chrome.runtime.onMessage.addListener((msg) => {
  if (msg?.type !== 'auth:change') return
  if (msg.event === 'SIGNED_OUT') location.href = chrome.runtime.getURL('auth/login.html')
  if (msg.event === 'SIGNED_IN')  refreshUi(msg.session)
})
```

## 미해결 항목 / 다음 단계

1. **OAuth 액션**: `auth.oauthLogin`(Google) 은 본 Phase 범위 밖. Phase 2.4 에서 `chrome.identity.launchWebAuthFlow` + state 검증 동반 구현. 라우팅 맵에 슬롯만 남김(현재 미등록).
2. **SW 재시작 시 구독 재등록**: `chrome.runtime.onStartup` 리스너에서도 `bootTasks` 를 호출하도록 추가 검토 필요. 현 코드는 onInstalled/초기 import 시점에만 실행.
3. **UMD 벤더 번들**: `lib/supabase-client.js` 가 `globalThis.supabase` 에 의존하므로 SW 최상단에 `import '../lib/vendor/supabase.umd.min.js'` 가 벤더링 완료 후 추가돼야 한다 (현재 service-worker.js 는 벤더 import 미포함 — Phase 1.3 summary 의 "UMD 벤더링" 지시 미이행 상태). **이 상태로는 실행 시 throw**.
4. **Rate limiting 방어**: login/signup 연속 실패 시 클라이언트 측 backoff 없음. 현재는 Supabase 서버 응답의 "rate limit" 에러를 `auth-error-map` 이 한국어로만 치환. Phase 2.3 에서 추가 고려.
