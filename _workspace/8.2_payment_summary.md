## 2. Edge Function 동작 모드

### 2-1. `mode=confirm` (클라이언트 주도)
Authorization Bearer JWT **필수**. 사용자가 외부 결제 랜딩에서 결제 성공 후 돌아올 때 호출.

```http
POST /functions/v1/verify-subscription
Authorization: Bearer <supabase_jwt>
Content-Type: application/json

{
  "mode": "confirm",
  "gateway": "toss",              // toss | portone
  "payment_id": "pay_2024xxxx",   // toss: paymentKey / portone: paymentId
  "plan": "pro",                  // pro | unlimited
  "orderId": "bm_abc_xxx",        // toss confirm API 용 (선택)
  "amount": 9900                  // toss confirm API 용 (선택, 서버 재검증됨)
}
```

흐름: JWT 검증 → PG API 로 결제 상세 조회 → **PG 가 알려준 실제 금액**과 `PLAN_PRICES` 비교 → 일치하면 `subscriptions` UPSERT (`onConflict: 'gateway,payment_id'`).

### 2-2. `mode=webhook` (PG 주도)
Authorization 헤더 **없음**. HMAC 서명 헤더 필수.

```http
POST /functions/v1/verify-subscription
x-toss-signature: sha256=abc...  (또는 x-portone-signature)
Content-Type: application/json

{ "eventType": "...", "paymentKey": "...", ... }
```

흐름: `rawBody` + `signatureHeader` + `{TOSS|PORTONE}_WEBHOOK_SECRET` 로 HMAC 검증 → `payment_id` 로 기존 subscriptions 행 조회 → **PG API 재조회**(원문 위변조 방지) → status 전이 (`refunded` / `expired` / `active`) 결정 → UPDATE.

### 2-3. 공통 응답 envelope
성공: `{ ok: true, data: ... }` / 실패: `{ ok: false, error: { code, message, details? } }` — edge_function_contracts §0.1 준수.

## 3. Idempotency 설계

| 경로 | 중복 방어 |
|---|---|
| confirm 재호출 | `UPSERT ... ON CONFLICT (gateway, payment_id) DO UPDATE` — UNIQUE 제약(005) 기준 |
| webhook 재전송 | subscriptions 행 SELECT → 현재 상태 == 계산된 nextStatus 이면 `changed: false` 로 조기 반환 |
| 토스 confirm "already confirmed" | TossError.message 에 `already` 포함 시 `fetchPayment` 폴백 |
| profiles.plan 이중 기록 | `refresh_user_plan` 의 `WHERE plan IS DISTINCT FROM new_plan` 조건으로 불필요 UPDATE 차단 |

### 6-2. PG 콘솔 설정

| 이벤트 | 처리 |
|---|---|

### 6-4. 테스트
- 토스 개발자센터 → Webhook 테스트 발송 → Edge Function 로그에서 `verify-subscription matched/changed` 확인.

## 7. DB 트리거(007) 요약

### 7-1. 객체 목록
- `plan_rank(text)` — free(0) < pro(1) < unlimited(2)
- `compute_effective_plan(uuid) → text`
- `refresh_user_plan(uuid)` — `profiles.plan = compute_effective_plan(user_id)`
- `subscriptions_sync_plan()` — AFTER INSERT/UPDATE/DELETE 트리거 함수
- `trg_subscriptions_sync_plan` — row-level 트리거
- `expire_due_subscriptions() → integer` — `active AND ends_at<NOW()` → `expired`

### 7-2. 만료 → free 복귀 체인
```
pg_cron → SELECT expire_due_subscriptions()
  → UPDATE subscriptions SET status='expired' WHERE ...
  → trg_subscriptions_sync_plan AFTER UPDATE
  → refresh_user_plan(user_id)
  → compute_effective_plan → 'free'
  → UPDATE profiles SET plan='free'
```

### 7-3. 권장 pg_cron 스케줄 (운영자 1회 실행)
```sql
-- Supabase Dashboard → Database → Extensions 에서 pg_cron 활성화 후:
SELECT cron.schedule(
  'expire-subscriptions',
  '*/15 * * * *',
  $$SELECT public.expire_due_subscriptions()$$
);
```

pg_cron 을 사용하지 않는 경우: Supabase Scheduled Functions 또는 외부 cron 이 `expire_due_subscriptions()` RPC 를 호출.

## 8. 확장 UI (checkout) 설계 결정

### 8-1. 외부 호스팅 랜딩 채택 (manifest CSP 최소화)

PG 위젯 스크립트 목록:
- `https://js.tosspayments.com/v2/standard` (토스)
- `https://cdn.portone.io/v2/browser-sdk.js` (포트원)

**문제:** MV3 기본 `extension_pages` CSP 는 `script-src 'self'` 로 원격 스크립트를 차단. CSP 완화는 심사 리스크 + 악성 CDN 체인 우려.

**선택:** `extension/payments/checkout.html` 은 플랜·수단 선택 UI만 담당하고, "결제 페이지로 이동" 클릭 시 `chrome.tabs.create({ url: EXTERNAL_BILLING_URL + query })` 로 **새 탭** 에서 외부 랜딩(예: `https://billing.blogbenchmarker.example/checkout`) 을 연다.

이점:
1. manifest CSP 변경 0, host_permissions 추가 0.
2. Chrome Web Store 심사 친화.
3. 3DS/간편결제 리디렉트, 결제 창 내 포커스 복원 등 UX 가 네이티브 브라우저 흐름.
4. PG 가 후속 webhook 을 보낼 때 이미 검증/등록된 public URL 이라 CORS 이슈 0.

### 8-2. 파라미터 전달
외부 랜딩으로 넘기는 쿼리스트링:
```
?plan=pro&gateway=toss&userId=<uuid>&orderId=bm_xxxxx&returnTo=chrome-extension
```
- orderId 는 클라이언트 생성(`crypto.getRandomValues`) — 서버가 필요 시 재발급 가능.
- userId 는 누수돼도 문제 없음(RLS 가 본인 격리). 다만 외부 랜딩은 별도 세션을 관리해야 하므로 실제 결제 확정은 반드시 JWT 기반 `verify-subscription/confirm` 을 거친다.

### 8-3. billing URL 해석
`resolveBillingUrl()` 이 `app_settings` 테이블의 `billing_url` 키를 읽어 동적 override. 없으면 `DEFAULT_BILLING_URL`. 운영자가 PG 별로 다른 랜딩을 운영할 때 코드 배포 없이 교체 가능.
